# P0.1 — Clean Project & Source Control (guided-build)

Concepts live in the **[P0.1 primer](P0.1-primer.md)** — project anatomy, target/module/plugin, the LFS rationale, folder/naming conventions. This plan does **not** repeat them; it's the buildable extension. Read the primer once first if it isn't fresh.

## Goal & design rationale
Stand up the single **umbrella practice project** — `UnifiedGameProject`, one UE 5.8 C++ game holding every later feature, created at this repo's root so planning and implementation share one repo — with source control that survives a binary-heavy, cross-device project. P1's code becomes the first production game's spine (GTFO-like, if green-lit at the P1 gate), so "clean" here is shipping-code hygiene, not busywork. Full rationale in the primer; the three drivers: **umbrella** (shared systems built once), **mobile from day one** (Android target + budget discipline now → the phase-gate is a formality), and **source control done right this time** (your EOS test `MultiplayerCourse` has a good `.gitignore` but **no LFS** and EOS code welded into the game module — both fixed here).

## Decisions locked for this plan (assumptions)
- **Template:** Blank C++ (keeps P0.4's first-person-character learning intact; avoids template cruft like the EOS test's `Variant_*` folders). If you'd rather strip a template, the create/strip steps below change — say so.
- **Module layout:** one primary game module now; EOS becomes a **plugin** in P0.2 (seam planned, not built here).
- **Project name & location:** `UnifiedGameProject` — decided. It names the module, targets, and Content namespace. Created as `UnifiedGameProject/` under this repo's root; **this repo already has git**, so P0.1 adds LFS + UE ignore/attribute files scoped to the project folder (no `git init`, no nested repo, no separate remote).

## Scope
**In:** fresh UE 5.8 Blank C++ project at `UnifiedGameProject/` (this repo's root); Git LFS added to this existing repo, UE `.gitignore`/`.gitattributes` scoped to the project folder; Content/Source structure + naming conventions; Project Settings baseline with **Windows + Android** target platforms; module layout decided with the P0.2 plugin seam documented.
**Out:** the EOS plugin itself (P0.2); Android SDK/NDK toolchain + real-device packaging (P0.5); any gameplay/content.
**Cut lines (if it busts 1 session, drop in this order):** (1) LFS file-*locking* setup; (2) the fine-grained Content subfolder tree (keep just `/Content/UnifiedGameProject/`). **Never cut:** `.gitignore` + `.gitattributes` LFS routing (verified working) + Android target set.

## Steps (1 session)

**A — Create & understand the project**
1. **[You]** Create a fresh **Blank C++** project named `UnifiedGameProject` in UE 5.8 (engine association `5.8`), with this repo's root as the location — UE creates the `UnifiedGameProject/` folder itself. *Why: first-of-a-kind — owning the create-a-C++-project flow is the point.*
2. **[You]** Open `Source/` and read each generated file once — `UnifiedGameProject.Target.cs`, `UnifiedGameProjectEditor.Target.cs`, `UnifiedGameProject/UnifiedGameProject.Build.cs`, module `.h/.cpp` — mapping them to the primer's anatomy table. *Why: internalizing target vs module is the core learning of this feature.*

**B — Source control (the core skill)**
3. **[Claude]** Draft `UnifiedGameProject/.gitignore` (current UE 5.8 set) and `UnifiedGameProject/.gitattributes` (LFS routing for `*.uasset *.umap *.fbx *.png *.tga *.wav …`), adapted from the EOS test's `.gitignore` + Epic's current recommended set. Both live *inside* the project folder so UE rules stay scoped to that subtree of this shared repo. Boilerplate.
4. **[You]** This repo already exists — **no `git init`** (a nested repo is exactly what we don't want). Run `git lfs install`; review both drafted files line-by-line so you know what each rule excludes/routes; run `git lfs track` from inside `UnifiedGameProject/` to confirm the patterns register. *Why: LFS is the new core skill — running it yourself is how it sticks.*
5. **[Pair]** Make the first commit, then **verify LFS actually worked**: `git lfs ls-files` lists your `.uasset`/`.umap`, and `git show HEAD:<some.uasset>` shows a *pointer* (version/oid/size), not binary. If a type slipped through, fix `.gitattributes` and re-add. *Integration + the "did it work" check.*
6. **[You]** Carry forward the EOS test's **secrets** pattern: gitignore the real platform `Config/Windows/WindowsEngine.ini` + `Config/Android/AndroidEngine.ini` (where EOS keys will live in P0.2) and plan matching `*.template` files. *Why: a security decision you must understand, not copy blindly.*

**C — Structure & settings**
7. **[Claude]** Create the `/Content/UnifiedGameProject/` namespace tree (`Characters/ Weapons/ Maps/ UI/ Environments/ Core/`) and drop a short naming-convention stub (`BP_ WBP_ SM_ SK_ T_ M_ MI_ DA_ A_`, aligned to ASSET-PIPELINE.md). Repetitive.
8. **[You]** Project Settings baseline: set **Windows + Android** as supported target platforms, default/startup maps, project name/description; confirm the single-module layout leaves clean room for the P0.2 plugin. *Why: this is where "mobile from day one" is wired and the plugin seam is verified.*

**D — Verify & close**
9. **[Pair]** Build the **Editor** target and launch; confirm a clean compile with no new warning/error spam in the log. Debug together if it complains.
10. **[You]** Create/confirm **this repo's** remote and push; confirm LFS objects upload (not committed as raw blobs). Then close the knowledge note (Claude drafts the mechanics sections; you add decisions/gotchas). *Why: closing the loop end-to-end, and the note is yours to return to.*
