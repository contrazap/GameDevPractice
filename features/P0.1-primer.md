# P0.1 — Clean Project & Source Control (primer)

> Primer only — concepts and decisions, **no step-by-step build**. When you're ready to build, invoke `guided-build` for P0.1 and it will extend this.

## Goal & design rationale
Create the single **umbrella practice project** — `UnifiedGameProject`, one UE 5.8 C++ game at this repo's root (planning and implementation share one repo) that every feature (all 17 reference games' mechanics) lives inside — plus source control that survives a binary-heavy, multi-device project. Everything in P0–P5 stands on this, and **P1's code is the production-candidate spine** (it carries into the GTFO-like game if green-lit at the P1 gate), so "clean" here pays off directly in shipping code.

Three rationales drive the choices:
- **Umbrella, not scattered demos.** Shared systems (input, saves, EOS, budgets) are built once and reused, and cross-cutting decisions (replication-aware, mobile-first) apply everywhere. A pile of separate template projects couldn't do that.
- **Mobile from day one.** Setting Windows **+ Android** as target platforms now, and authoring to mobile budgets (CONVENTIONS working rule #1), is what turns the per-phase Android gate into a formality instead of a rewrite.
- **Do source control right *this time*.** Your EOS test (`MultiplayerCourse`) has a good `.gitignore` but **no `.gitattributes` / no Git LFS** — binary `.uasset`/`.umap` files are sitting in plain git, where every save stores a full new copy and bloats history permanently. P0.1 fixes this *before* content accumulates. It also had EOS code living *inside* the game module (not reusable) — P0.1 sets up module layout so P0.2 can extract EOS as a portable plugin.

## Primer — concepts & engine systems

### 1. Project anatomy (what a UE C++ project actually is)
A `.uproject` is a small JSON manifest: engine association (`"5.8"`), the list of **Modules**, and enabled **Plugins**. The real code lives under `Source/`. Key files and what each is *for*:

| File | Role |
|---|---|
| `UnifiedGameProject.uproject` | Manifest: engine version, modules, plugins. |
| `Source/UnifiedGameProject.Target.cs` | Build **target**: the *Game* build (packaged). Defines type + included modules. |
| `Source/UnifiedGameProjectEditor.Target.cs` | The *Editor* build (what you run in-editor). Later you may add Client/Server targets. |
| `Source/UnifiedGameProject/UnifiedGameProject.Build.cs` | The **module's** rules: public/private module **dependencies** (Engine, EnhancedInput, OnlineSubsystem, …). |
| `Source/UnifiedGameProject/UnifiedGameProject.h/.cpp` | Module implementation (`IMPLEMENT_PRIMARY_GAME_MODULE`). |

Mental model: **Targets** = *what* to build (Editor vs packaged Game, for which platform/config). **Modules** = units of C++ that compile into a target and declare their dependencies. **Plugins** = self-contained, portable bundles of modules+content with their own `.uplugin`. This target/module/plugin distinction is the single most useful thing to internalize here — it's what makes P0.2's EOS plugin reusable across every production game.

### 2. Module vs Plugin (the P0.2 setup decision)
- A **module** compiles into *this* project. Fine for game-specific code.
- A **plugin** is portable: drop its folder into any project, enable it in the `.uproject`, done. It can gate by platform/target (`TargetAllowList`) and ship independently.
- Your EOS test put `EOSSessionSubsystem` directly in the `MultiplayerCourse` game module — coupled, not reusable. **P0.1's job:** stand up one clean **primary game module** now, and *plan the seam* so P0.2 adds EOS as a `Plugins/<Name>/` plugin with a small API surface and zero gameplay code. You don't build the plugin in P0.1 — you just don't paint yourself into a corner.

### 3. Source control for binary-heavy projects (the core skill)
This planning repo already exists and the project lives inside it — so **no `git init`** (a nested repo is exactly what we don't want). Two files do the work, both placed *inside* `UnifiedGameProject/` (nested git ignore/attribute files apply to their own subtree, keeping UE rules scoped while the planning files at the repo root stay plain):
- **`.gitignore`** — exclude generated/rebuildable dirs: `Binaries/`, `Intermediate/`, `Saved/`, `DerivedDataCache/`, `.vs/`, generated `*.sln`. **Track** `Content/`, `Config/`, `Source/`, `.uproject`. (Your EOS test's `.gitignore` is a solid starting point — it also has a good **secrets** pattern: ignore the real platform `WindowsEngine.ini`/`AndroidEngine.ini` that hold EOS keys, commit only `*.template`. Carry that forward — EOS + Android are coming.)
- **`.gitattributes` + Git LFS** — the piece the EOS test is *missing*. Route binary asset types through LFS so git stores a pointer, not the bytes, in history:
  ```gitattributes
  *.uasset  filter=lfs diff=lfs merge=lfs -text
  *.umap    filter=lfs diff=lfs merge=lfs -text
  *.fbx     filter=lfs diff=lfs merge=lfs -text
  *.png     filter=lfs diff=lfs merge=lfs -text
  *.tga     filter=lfs diff=lfs merge=lfs -text
  *.wav     filter=lfs diff=lfs merge=lfs -text
  ```
  Why it matters: `.uasset`/`.umap` are **binary and unmergeable** — two people editing the same asset can't auto-merge. LFS keeps history lean; **file locking** (`git lfs lock`) is the discipline that prevents lost work on shared assets. Solo now, but it's a real-project habit and cheap to set up.

### 4. Folder & naming conventions (decide once, obey forever)
- **Content:** namespace everything under a top-level `/Content/UnifiedGameProject/` folder (so later plugin/migrated content never collides), with subfolders like `Characters/ Weapons/ Maps/ UI/ Environments/ Core/`. Optional `/Content/Developers/` for scratch.
- **Asset naming:** follow ASSET-PIPELINE.md — `SM_` static mesh, `SK_` skeletal, `T_` texture (`_D/_N/_ORM`), `M_`/`MI_` materials — plus `BP_` blueprint, `WBP_` widget, `DA_` data asset, `A_` anim.
- **Source:** `Source/UnifiedGameProject/` for the primary module; keep gameplay in sensibly grouped subfolders from the start.

### Architecture sketch (target repo layout)
```
GameDevPractice/                    (this repo — planning + implementation together)
├─ ROADMAP.md · phases/ · features/ · knowledge/ · …   (the planning system, plain text)
└─ UnifiedGameProject/              (the umbrella UE project — created in P0.1)
   ├─ UnifiedGameProject.uproject   modules + plugins manifest
   ├─ .gitignore                    ignore Binaries/ Intermediate/ Saved/ DDC/ .vs/ *.sln (applies to this subtree)
   ├─ .gitattributes                LFS routing for *.uasset *.umap *.fbx textures audio (applies to this subtree)
   ├─ Config/                       Default*.ini (+ Windows/ Android/ platform configs)
   │   └─ (real EOS-key .ini gitignored; *.template committed)   ← from P0.2
   ├─ Content/
   │   └─ UnifiedGameProject/       Characters/ Weapons/ Maps/ UI/ Core/ …
   ├─ Source/
   │   ├─ UnifiedGameProject.Target.cs         Game target

```

## Key decisions to make before building
1. **Starting template.** Options in 5.8 (all present locally): **Blank C++**, **TP_FirstPerson C++**, **TP_ThirdPerson C++**. Recommendation: **Blank C++**. The first production candidate is first-person and P0.4 builds the FP character as a **[You]** learning task — starting from a template that ships a character would pre-empt that learning and leave stripping work. Blank keeps P0.4 meaningful and the repo genuinely clean. (Your EOS test used the combined Third Person template, which is why it carried `Variant_Combat/Platforming/SideScrolling` cruft.)
2. **Project name — decided: `UnifiedGameProject`.** It names the module, targets, and Content namespace, and is created as `UnifiedGameProject/` at this repo's root so planning and implementation live in one repo.
3. **Module layout.** Single primary game module now; EOS as a **plugin** in P0.2. (Alternative — a separate "Core" gameplay module — is overkill for a practice project; plugins give you the reuse you actually want.)

## Scope
**In:** fresh UE 5.8 Blank C++ project at `UnifiedGameProject/` (this repo's root); Git LFS added to this existing repo, UE `.gitignore`/`.gitattributes` scoped to the project folder; Content/Source folder structure + naming conventions; project settings baseline with **Windows + Android** target platforms; module layout decided with a planned seam for the EOS plugin.
**Out:** the EOS plugin itself (P0.2); Android SDK/NDK toolchain + actual device packaging (P0.5); any gameplay.
**Cut line if it busts 1 session:** defer LFS *file-locking* setup and the fine-grained Content subfolder tree; keep `.gitignore` + `.gitattributes` LFS routing + Android target — those are the non-negotiables.

- **Networking:** n/a to build here, but the *structural* decision matters — a clean primary module + EOS-as-plugin is what lets every later feature be server-authoritative without refactoring. No 2-client smoke test this feature.
