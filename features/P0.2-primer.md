# P0.2 — EOS Plugin Port (primer)

> Primer only — concepts and decisions, no step-by-step build. The build plan is
> [P0.2-eos-plugin-port.md](P0.2-eos-plugin-port.md); it extends this file and doesn't repeat it.

## Goal & design rationale

Extract the working EOS multiplayer from the test project (`MultiplayerTestEOS`, snapshotted
in-repo at [reference/eos/](../reference/eos/) — Epic login, host/join/leave lobby sessions,
RTC voice + the receive-bug workaround) into a **portable plugin** in `UnifiedGameProject`, with proper
separation this time: no gameplay code in the plugin, a small clean API the game calls.

Why this matters beyond practice:
- **Every production game ships this plugin.** The ecosystem model (coop + episodic DLC, cross-game
  progression) means sessions/login/voice are shared substrate — build once, drop into each title.
  P0.6 will add EOS Player Data Storage (cloud saves) to the *same* plugin, so name and scope it
  account-level, not session-only.
- **The test project proved the tech but welded it in.** `UEOSSessionSubsystem` sits inside the
  `MultiplayerCourse` game module with a hardcoded travel map, a hardcoded bucket ID, and UI concerns
  nearby. Porting it "properly" is the code-organization rep this feature exists for.
- **Second EOS exposure = retention.** You built this once with heavy assistance; re-walking login →
  lobby → P2P travel → voice while restructuring it is deliberate spaced repetition.

## Primer — concepts & engine systems

### 1. Plugin anatomy (the new material)

A plugin is a self-contained folder under `Plugins/` with a `.uplugin` descriptor (JSON, like a
mini-`.uproject`) plus its own `Source/`, and optionally `Content/`, `Config/`, `Resources/`.
Verified against the installed 5.8 engine's own `OnlineSubsystemEOS.uplugin`:

| Descriptor field | What it does |
|---|---|
| `Modules[]` | The plugin's code modules: `Name`, `Type` (`Runtime` for us), `LoadingPhase` (`Default` is fine — subsystems register at module load), `PlatformAllowList` (e.g. `["Win64","Android"]` — other platforms skip compiling it) |
| `Plugins[]` | **Plugin-to-plugin dependencies** — our plugin declares it needs `OnlineSubsystem`, `OnlineSubsystemUtils`, `OnlineSubsystemEOS`, `VoiceChat` here |
| `EnabledByDefault`, `Category`, `Description` | Metadata; keep `EnabledByDefault: false`, enable per-project in the `.uproject` |

Folder layout mirrors a game module:
```
Plugins/OnlineCore/
├─ OnlineCore.uplugin                  descriptor: modules + plugin deps + platforms
├─ README.md                         drop-in guide (the reusability contract)
├─ Config/                           (maybe — see Verify list)
└─ Source/OnlineCore/
   ├─ OnlineCore.Build.cs              module deps (link-level)
   ├─ Public/                        the API the game may include  ← keep SMALL
   └─ Private/                       everything else (module impl, workaround guts)
```
The module's API macro becomes `ONLINECORE_API` (replaces `MULTIPLAYERCOURSE_API`). Creation route:
editor plugin wizard (Edit → Plugins → Add, "Blank" template) or hand-rolled — both end at the same
files; hand-rolling teaches more and is ~5 files.

### 2. The two-level dependency model (most common plugin confusion)

Dependencies are declared **twice, at different levels**, and both are required:
- **`.uplugin → Plugins[]`**: "when I'm enabled, these plugins must be enabled" (plugin graph).
- **`Build.cs → Public/PrivateDependencyModuleNames`**: "my C++ links against these modules"
  (linker graph). `Public` if a dependency's types appear in your public headers, `Private` otherwise
  — our subsystem's *public* header needs `OnlineSubsystem`/`OnlineSessionSettings` types (public),
  while `OnlineSubsystemEOS` + `VoiceChat` are implementation details (private).

Miss the descriptor entry and the module may not even be loaded when yours compiles; miss the
Build.cs entry and you get linker errors. Verified: `IOnlineSubsystemEOS.h` (needed by the voice
workaround for `GetVoiceChatUserInterface`) is a **Public** header of `OnlineSubsystemEOS`, so the
workaround compiles fine from an external plugin with a private dependency.

### 3. Why the subsystem needs zero glue code

`UEOSSessionSubsystem` is a `UGameInstanceSubsystem`. Subsystems are **auto-discovered**: the engine
instantiates every `UGameInstanceSubsystem` subclass found in any loaded module when the
GameInstance initializes. Move the class into the plugin, enable the plugin — it exists at runtime,
no game-side registration, no custom GameInstance. This is what makes "drop the folder into any
project" literally true, and it's why the game reaches it the same way the test did:
`GetGameInstance()->GetSubsystem<UEOSSessionSubsystem>()`.

### 4. Separation design — what moves, what stays (the core [You] thinking)

| Concern | Where | Why |
|---|---|---|
| Login/logout, host/find/join/leave, session state queries | **Plugin** (public API) | The reusable substrate |
| Voice mute/unmute + the RTC receive-bug workaround | **Plugin** | EOS-layer, not gameplay; every game needs it |
| Android mic-permission request | **Plugin** (at login/voice setup) | It exists *because of* EOS voice — test project wrongly put it in the PlayerController |
| Travel map URLs (`?listen` map, return-to-menu map) | **Game** → passed as parameters / project config | The #1 hardcoding sin in the test: `/Game/ThirdPerson/Lvl_ThirdPerson?listen` baked into the subsystem |
| Session bucket ID (`GTFOClone_v1`) | **Game** → per-project config | Bucket partitions lobbies per game/version — by definition not plugin code |
| Exec commands, session menu (UMG over a C++ widget base — CONVENTIONS rule 6), voice HUD | **Game** module | UI is gameplay-adjacent; plugin stays headless. The test's raw-Slate menu was zero-content debug expedience, not the production pattern |
| On-screen debug messages (`ShowMsg`) | Plugin, but as log + optional delegate | Screen spam is a debug choice the game should own — cheap cleanup, decide during the build |

**API surface options** for the game-owned values, in increasing effort: (a) plain function
parameters (`HostSession(ListenMapUrl)`), (b) a `UDeveloperSettings` config class in the plugin
(`UOnlineCoreSettings`: BucketId, default return map — appears in Project Settings, per-game ini),
(c) BlueprintAssignable state-change delegates (`OnLoginStateChanged`, `OnSessionStateChanged`) so
UI binds instead of polling. Recommendation: **(a)+(b) required, (c) if time allows** — the test's
polling UI worked fine at this scale.

### 5. Config layering & secrets (P0.1 pattern activates now)

The plugin carries *code*; EOS still needs **per-project engine config** that cannot live inside the
plugin: `[OnlineSubsystem] DefaultPlatformService=EOS`, the `NetDriverEOS` NetDriverDefinitions
block, and `[/Script/OnlineSubsystemEOS.EOSSettings]`. The split (from the test project, keep it):
- **Committed** `Config/DefaultEngine.ini` — all non-secret EOS blocks (scopes, RTC, net driver).
- **Gitignored** `Config/Windows/WindowsEngine.ini` + `Config/Android/AndroidEngine.ini` — the
  `+Artifacts=(ClientId=…,ClientSecret=…)` line. P0.1 already wrote the ignore rules for exactly
  these paths; committed `*.template` copies document the shape. Platform inis layer over Default
  and **do** cook into packaged builds — that's why the pattern works.
- The plugin's README must state this contract: "enabling the plugin is not enough; add these ini
  blocks + your portal credentials."

Portal side: the dedicated **UnifiedGameProject product** (created 2026-07-11, same Epic
organization as GTFOClone — existing member invites carry over). `DefaultArtifactName=UnifiedGameProject`;
bucket `UnifiedGameProject_v1`. A fresh product must be *verified, not trusted* — client policy
**Peer2Peer + Voice**, EAS application with **Basic Profile only** and the client linked; the build
plan opens with this checklist. One field the portal does **not** issue: **`ClientEncryptionKey`**,
a self-generated 64-hex string the SDK uses to client-side-encrypt Player Data Storage uploads
(P0.6 cloud saves). Nothing in P0.2 uses it, but it must stay **stable for life**: change or lose it
and previously uploaded cloud saves become undecryptable — login/sessions/voice/local saves are
unaffected; a regenerated key just restarts cloud saves from zero. Generate once in P0.2, store it
in your password manager, fill it into the gitignored platform inis alongside the ClientSecret.

### 6. EOS flow review (second exposure — pointers, not re-teaching)

One paragraph to re-anchor: `Login()` → Epic **account portal** (or `persistentauth` cache, or
`-AUTH_TYPE=developer` + DevAuthTool for local two-account iteration) → `HostSession()` creates an
EOS **lobby** (`bUseLobbiesIfAvailable`, bucket attribute, RTC voice room auto-attached) and
`ServerTravel(map?listen)` → clients `FindSessions` (filter: `SEARCH_LOBBIES` + same bucket) →
`JoinSession` → `ClientTravel` to the resolved `EOS:<PUID>:GameNetDriver:…` P2P address. Voice joins
automatically; the workaround re-applies receive options (`SetPlayerVolume(name, 1.0)`) after join
storms because 5.8's `EOSVoiceChat` leaves late joiners unsubscribed. Full detail lives in-repo
at `reference/eos/EOS_Setup_Guide/` (chapters 03 sessions, 09 voice, 10 menu/HUD) and the
verified-facts table in `reference/eos/EOS_MULTIPLAYER_SETUP.md` §1 — consult those, don't rebuild
from memory.

**Testing model (this bites people):** multi-client **PIE cannot test EOS** — one process = one EOS
instance = one login. The working smoke is **two standalone instances + DevAuthTool** (two Epic
accounts as `Cred1`/`Cred2` on port 8081). The build plan swaps the standard 2-client PIE smoke for
this, per CONVENTIONS rule 2's intent.

### Architecture sketch

```
UnifiedGameProject.uproject ── enables ──► OnlineCore.uplugin ── declares ──► OnlineSubsystemEOS,
                                                │                            VoiceChat, OSS, OSSUtils
Game module (UnifiedGameProject)                │ Source/OnlineCore
  DebugPlayerController (exec + menu UI)        │   Public/  EOSSessionSubsystem.h   ← the API
  calls GetSubsystem<UEOSSessionSubsystem>() ──►│   Private/ subsystem impl + voice workaround
  passes: maps, reads: state, binds: delegates  │
                                                │
Config/DefaultEngine.ini (committed: EOS blocks, NetDriverEOS)
Config/<Platform>/<Platform>Engine.ini (gitignored: +Artifacts secrets)   ← templates committed
```

   P1.11 lobby flow hardens it. Name verified free against the installed 5.8 engine's plugins/modules.
