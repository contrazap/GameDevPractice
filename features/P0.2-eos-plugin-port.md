## Mental model

The five ideas that organize everything here:

1. **A plugin is a folder with a passport.** `Plugins/OnlineCore/` + `OnlineCore.uplugin` (JSON
   descriptor). The engine *discovers* it — there is no IDE "add project" step; VS solutions are
   generated output. Anatomy in Session 1 Teach.
2. **Dependencies are declared twice, at two levels.** `.uplugin → Plugins[]` = plugin graph
   ("when I'm enabled, enable these"); `Build.cs → Public/PrivateDependencyModuleNames` = linker
   graph ("my C++ links these"). Miss the first → module may not be loaded; miss the second →
   linker errors. Public only if the dependency's types appear in your public headers.
3. **Subsystems self-register.** `UEOSSessionSubsystem : UGameInstanceSubsystem` is auto-discovered
   in any loaded module when the GameInstance initializes — zero glue code, which is what makes
   "drop the folder into any project" literally true. The game reaches it via
   `GetGameInstance()->GetSubsystem<UEOSSessionSubsystem>()`.
4. **The port is a separation exercise.** Sessions/voice/mic-permission → plugin; maps, bucket,
   UI → game (parameters + `UOnlineCoreSettings`). The what-moves table (Session 1 Teach) is the
   core [You] thinking.
5. **EOS flow, second exposure:** `Login()` → Epic account portal / `persistentauth` cache /
   `-AUTH_TYPE=developer` + DevAuthTool → `HostSession()` creates an EOS **lobby**
   (`bUseLobbiesIfAvailable`, bucket attribute, RTC voice room auto-attached) +
   `ServerTravel(map?listen)` → clients `FindSessions` (`SEARCH_LOBBIES` + same bucket) →
   `JoinSession` → `ClientTravel` to the resolved `EOS:<PUID>:GameNetDriver:…` P2P address. Voice
   joins automatically; the workaround re-applies receive options (`SetPlayerVolume(name, 1.0)`)
   after join because 5.8's `EOSVoiceChat` leaves late joiners unsubscribed. Full detail:
   [EOS_Setup_Guide](../reference/eos/EOS_Setup_Guide/) ch. 03/09/10 + the verified-facts table in
   [EOS_MULTIPLAYER_SETUP.md](../reference/eos/EOS_MULTIPLAYER_SETUP.md) §1.

```
UnifiedGameProject.uproject ── enables ──► OnlineCore.uplugin ── declares ──► OnlineSubsystemEOS,
                                                │                            VoiceChat, OSS, OSSUtils
Game module (UnifiedGameProject)                │ Source/OnlineCore
  PlayerController (exec + menu UI)             │   Public/  EOSSessionSubsystem.h   ← the API
  calls GetSubsystem<UEOSSessionSubsystem>() ──►│   Private/ subsystem impl + voice workaround
  passes: maps · reads: state                   │
Config/DefaultEngine.ini            (committed: EOS blocks, NetDriverEOS)
Config/<Platform>/<Platform>Engine.ini (gitignored: +Artifacts secrets — templates committed)
```

## Session 1 — plugin skeleton & the port

### Teach

**Plugin anatomy.** A plugin = folder under `UnifiedGameProject/Plugins/` + `.uplugin` descriptor +
its own `Source/`, optionally `Content/`, `Config/`, `Resources/`. Target layout:

```
Plugins/OnlineCore/
├─ OnlineCore.uplugin                descriptor: modules + plugin deps + platforms
├─ README.md                         drop-in guide (Session 3 — the reusability contract)
└─ Source/OnlineCore/                ← folder name must equal the module name
   ├─ OnlineCore.Build.cs            module deps (link level)
   ├─ Public/                        the API the game may include — keep SMALL
   └─ Private/                       everything else (module impl, workaround guts)
```

Descriptor fields, verified against the engine's own
`C:\Program Files\Epic Games\UE_5.8\Engine\Plugins\Online\OnlineSubsystemEOS\OnlineSubsystemEOS.uplugin`:

| Field | What it does |
|---|---|
| `FileVersion` | **Mandatory, value 3** — descriptor parsing hard-fails without it (`PluginDescriptor.cpp` reads it first and rejects the file otherwise) |
| `Modules[]` | The plugin's code modules: `Name`, `Type` (`Runtime` for us), `LoadingPhase` (`Default` — subsystems register at module load), `PlatformAllowList` (`["Win64","Android"]` — other platforms skip compiling it) |
| `Plugins[]` | Plugin-to-plugin dependencies: `OnlineSubsystem`, `OnlineSubsystemUtils`, `OnlineSubsystemEOS`, `VoiceChat` |
| `EnabledByDefault`, `Category`, `Description` | Metadata; keep `EnabledByDefault: false` — enable per-project in the `.uproject` (the reusable-plugin contract) |

**Names must agree.** Module `"Name"` in the `.uplugin` = `Source/<Name>/` folder =
`<Name>.Build.cs` filename = `class <Name> : ModuleRules` inside it = second argument of
`IMPLEMENT_MODULE`. Any mismatch → module not found, or a name-mismatch assert at startup.

**How creation actually works (no wizard, no VS "Add Project").** Files on disk, then: close the
editor → create/edit files → right-click `UnifiedGameProject.uproject` → *Generate Visual Studio
project files* → build Development Editor | Win64 → open editor. The wizard (Edit → Plugins → +
Add) would scaffold all of this — deliberately unused; the anatomy is the learning. Build-order
gotchas: an **empty/invalid Build.cs breaks project-file generation** (don't regenerate before
step 2 is written), and UBT only compiles **enabled** plugins — with `EnabledByDefault: false`
nothing builds until the `.uproject` lists it, so that one line gets pulled forward into step 3.

**The module class — what `IMPLEMENT_MODULE` is.** Each module is a DLL in editor builds and
statically linked in packaged builds. `FModuleManager` needs a factory it can invoke by name;
`IMPLEMENT_MODULE(FOnlineCoreModule, OnlineCore)` emits the correct registration for **both** link
modes and static-asserts the name matches the module. Exactly one per module, in a `.cpp`. Our
subsystem self-registers (Mental model #3), so empty lifecycle bodies are *correct*, not lazy —
the class exists as the place P0.6 will later hang startup work. Complete skeleton:

```cpp
// Public/OnlineCore.h
#pragma once
#include "Modules/ModuleManager.h"

class FOnlineCoreModule : public IModuleInterface
{
public:
    virtual void StartupModule() override;
    virtual void ShutdownModule() override;
};

// Private/OnlineCore.cpp
#include "OnlineCore.h"

void FOnlineCoreModule::StartupModule() {}
void FOnlineCoreModule::ShutdownModule() {}

IMPLEMENT_MODULE(FOnlineCoreModule, OnlineCore)
```

No `ONLINECORE_API` on the module class — nothing outside the module constructs it; the macro
starts mattering on the subsystem (step 3). Engine examples: named-class shape →
`C:\Program Files\Epic Games\UE_5.8\Engine\Plugins\Online\EOSShared\Source\EOSShared\Private\EOSSharedModule.h`;
3-line stock-impl alternative →
`C:\Program Files\Epic Games\UE_5.8\Engine\Plugins\Online\OnlineSubsystemUtils\Source\OnlineBlueprintSupport\Private\OnlineBlueprintSupportModule.cpp`.

**From `#include` to module name — how a Build.cs is actually derived (step 2's real skill).**
An include list alone won't tell you the modules; you must resolve each header to its **owner**:

1. Collect includes from **both** the `.h` and the `.cpp` — they land in different lists.
2. Locate each header on disk: in VS, cursor on the `#include` → **F12** (or right-click → *Open
   Document*); the opened tab's path is the answer.
3. Read the module off the path — Unreal's folder layout *is* the module registry:

| Path shape | Module |
|---|---|
| `Engine/Source/Runtime/<M>/Public\|Classes/…` | `<M>` |
| `Engine/Plugins/…/<Plugin>/Source/<M>/Public/…` | `<M>` |
| `Engine/Plugins/…/<Plugin>/Source/Public/…` | the plugin's single module |

**Public vs Private.** Included by *our* public header → it's part of our API, downstream code must
resolve those types → `PublicDependencyModuleNames`. Included only in `.cpp`/private code →
`PrivateDependencyModuleNames`. Over-declaring Public isn't a build break; it leaks our
dependencies' include paths onto the game module.

**Two deps no `#include` reveals:** `CoreUObject` (nothing names it — `UCLASS`/`GENERATED_BODY`/
`.generated.h` *are* its machinery; reflection markup ⇒ CoreUObject) and `AndroidPermission` (not in
this file at all — it arrives with the mic-permission code moving in from the test's
PlayerController; audit **all code that will live in the module**, not just the file you're reading).

**What moves, what stays (the core [You] thinking for steps 3–4):**

| Concern | Where | Why |
|---|---|---|
| Login/logout, host/find/join/leave, state queries | **Plugin** (public API) | The reusable substrate |
| Voice mute/unmute + RTC receive-bug workaround | **Plugin** | EOS-layer, not gameplay |
| Android mic-permission request | **Plugin** (voice setup path) | Exists *because of* EOS voice — the test wrongly put it in the PlayerController |
| Travel map URLs (`?listen` map, return map) | **Game** → parameters / config | The #1 hardcoding sin in the test |

**Config as API — `UDeveloperSettings` (step 4).** The bucket ID and return map are *the game's*
choices, so the plugin must not hardcode them — but it can't take them as function arguments either
(nothing would pass them). `UDeveloperSettings` is the engine's answer: a UObject whose properties
appear automatically in **Project Settings** and persist to an ini. Like subsystems, it
self-registers — declare the class and it shows up, no glue. Skeleton:

```cpp
// Public/OnlineCoreSettings.h
#pragma once
#include "Engine/DeveloperSettings.h"
#include "OnlineCoreSettings.generated.h"

UCLASS(config = Game, defaultconfig, meta = (DisplayName = "Online Core"))
class ONLINECORE_API UOnlineCoreSettings : public UDeveloperSettings
{
	GENERATED_BODY()
public:
	/** Partitions EOS lobbies — only sessions sharing this bucket are found. */
	UPROPERTY(EditAnywhere, config, Category = "Session")
	FString BucketId = TEXT("UnifiedGameProject_v1");

	UPROPERTY(EditAnywhere, config, Category = "Session")
	FString DefaultReturnMap = TEXT("/Game/Maps/L_Sandbox");
};
```

Read it anywhere with `GetDefault<UOnlineCoreSettings>()->BucketId`. Four traps:
- **`config` appears twice** — on the `UCLASS` (`config = Game`) *and* on every `UPROPERTY` that
  must persist. Omit it on a property and the value silently resets each run.
- `defaultconfig` writes to `Config/DefaultGame.ini`, section `[/Script/OnlineCore.OnlineCoreSettings]`
  — the pattern is `/Script/<ModuleName>.<ClassName without the U>`.
- **`Build.cs` must gain `DeveloperSettings`** (public — the base class is in *our* public header).
  Nothing else pulls it in; without it you get an unresolved-include error.
- Engine example:
  `C:\Program Files\Epic Games\UE_5.8\Engine\Source\Runtime\Engine\Classes\Engine\AssetManagerSettings.h:66`.

**The mic-permission move (step 4).** Source to lift, with its guard:
[MultiplayerCoursePlayerController.cpp:28-30, 58-64](../reference/eos/src-snapshot/MultiplayerCoursePlayerController.cpp) —

```cpp
#if PLATFORM_ANDROID
#include "AndroidPermissionFunctionLibrary.h"    // top of file, inside the guard
#endif
...
#if PLATFORM_ANDROID
	if (!UAndroidPermissionFunctionLibrary::CheckPermission(TEXT("android.permission.RECORD_AUDIO")))
	{
		UAndroidPermissionFunctionLibrary::AcquirePermissions({ TEXT("android.permission.RECORD_AUDIO") });
	}
#endif
```

That `#if PLATFORM_ANDROID` is exactly why `AndroidPermission` is a **private, Android-only** dep in
the Build.cs — on Win64 the code doesn't exist, so the module isn't linked. It belongs in the
plugin's voice-setup path (it exists *because of* EOS voice), not in a PlayerController's `BeginPlay`.

**Config & secrets (steps 5–6, the P0.1 pattern activates).** The plugin carries *code*; EOS still
needs per-project engine config that cannot live inside it: `DefaultPlatformService=EOS`, the
`NetDriverEOS` definitions, `[/Script/OnlineSubsystemEOS.EOSSettings]`. Split: **committed**
`Config/DefaultEngine.ini` (all non-secret blocks) vs **gitignored**
`Config/Windows|Android/<Platform>Engine.ini` (the `+Artifacts=(ClientId=…,ClientSecret=…)` line;
committed `*.template` copies document the shape). Platform inis layer over Default and **do** cook
into packaged builds — that's why the pattern works. One field the portal does **not** issue:
`ClientEncryptionKey` — self-generated 64 hex chars, used to client-side-encrypt P0.6's cloud
saves. Nothing in P0.2 uses it, but it must stay **stable for life** (a changed key makes uploaded
saves undecryptable; everything else keeps working). Generate once, store in the password manager.

**Trap 1 — `HostSession` gets the URL *raw*.** The host's travel is
`ServerTravel(PendingListenMapUrl)`
([EOSSessionSubsystem.cpp:181](../UnifiedGameProject/Plugins/OnlineCore/Source/OnlineCore/Private/EOSSessionSubsystem.cpp#L181))
— the plugin appends **nothing**. Pass the map path *including* the option:
`/Game/UnifiedGameProject/Maps/L_Sandbox?listen`. Drop `?listen` and you travel to the map as a
standalone server with a live EOS session nobody can connect to — a failure that looks like "join
finds the session but the connection times out".

**Trap 2 — the plugin is headless, so by default you see nothing.** Session 1 demoted `ShowMsg` to
`UE_LOG(LogOnlineCore)` + `OnMessage.Broadcast(Msg, Color)`
([EOSSessionSubsystem.cpp:406](../UnifiedGameProject/Plugins/OnlineCore/Source/OnlineCore/Private/EOSSessionSubsystem.cpp#L406)).
That was the right call — and it means **the game must bind `OnMessage`** or the whole flow runs
silently. The PlayerController binds it and calls `AddOnScreenDebugMessage`, which is exactly the
"screen output is the game's choice" contract being honoured rather than described. `OnMessage` is a
**dynamic** delegate → the handler must be a `UFUNCTION()` and is bound with `AddDynamic`.

**The map is already configured.**
[Config/DefaultGame.ini](../UnifiedGameProject/Config/DefaultGame.ini) carries
`BucketId=UnifiedGameProject_v1` and
`DefaultReturnMap=/Game/UnifiedGameProject/Maps/L_Sandbox` under

**UMG: the `BindWidget` contract** (first encounter). In C++:

```cpp
UPROPERTY(meta = (BindWidget)) TObjectPtr<UButton> HostButton;
```

The WBP deriving this class **must** contain a widget of matching **type and exact name**, or it
refuses to compile in the UMG editor. That error is not friction — it *is* the contract: the C++
declares the parts it needs, the designer supplies them, and a typo is caught at asset-compile time
instead of as a null pointer at runtime. There is no wiring step and no `FindWidget` call; the
engine populates the pointer before `NativeConstruct`. Widgets you want but may not have get
`meta = (BindWidgetOptional)`.

Lifecycle (verified `UserWidget.h:1584-1586`): `NativeConstruct()` — bind click handlers here, the
BindWidget pointers are live. `NativeTick(const FGeometry&, float)` — poll state here.
`NativeDestruct()` — unbind. Clicks: `UButton::OnClicked` is
`DECLARE_DYNAMIC_MULTICAST_DELEGATE(FOnButtonClickedEvent)` — **zero params** (`Button.h:18,76`), so
handlers are `UFUNCTION() void OnHostClicked();` bound with
`HostButton->OnClicked.AddDynamic(this, &USessionMenuWidgetBase::OnHostClicked)`. Label flipping:
`UTextBlock::SetText(FText)` (`TextBlock.h:103`). Guards: `UWidget::SetIsEnabled(bool)`
(`Widget.h:545`).

**Why poll instead of subscribe.** The subsystem broadcasts messages but has no state-change
delegate (cut line 1 — deliberately unbuilt). So the menu re-reads `IsLoggedIn()` / `IsInSession()` /
`IsVoiceActive()` every `NativeTick` and pushes the result into `SetIsEnabled` / `SetText`. This is
exactly what the test project's Slate `TAttribute` lambdas did every frame
([10_InGame_Menu_Voice_HUD.md §3](../reference/eos/EOS_Setup_Guide/10_InGame_Menu_Voice_HUD.md)) —
same cost, same behaviour, and it keeps the plugin's surface small. The state-guard table to
reproduce:

| Button | Enabled when | Label |
|---|---|---|
| Login/Logout | logged out: always · logged in: **only when not in a session** | flips on `IsLoggedIn()` |
| Host · Join | `IsLoggedIn() && !IsInSession()` | — |
| Leave | `IsInSession()` | — |
| Mic · Speaker | `IsVoiceActive()` | `Mic: ON` / `Mic: MUTED` |
| Resume · Quit | always | — |

**Input mode — the mouse-look fix** (carried from ch.10 §2; the *behaviour* transfers, the Slate does
not). The PlayerController, not the widget, owns this:

| Menu state | Input mode | Cursor |
|---|---|---|
| closed (playing) | `FInputModeGameOnly` | hidden — direct mouse-look |
| open | `FInputModeGameAndUI` | shown — buttons clickable, **Escape still reaches the PC** |

`UIOnly` would starve the PlayerController's input component and Escape could no longer close the
menu. Bind the toggle directly (`EKeys::Escape`, `EKeys::Android_Back` — not in any Input Mapping
Context, so a plain `BindKey` in `SetupInputComponent` is simplest). **Never `SetPause`** — this is
multiplayer; one client pausing must not touch anyone else. One PIE reality: **the editor's
Stop-Play binding eats Escape** before the game ever sees the key, so the controller also binds
**`M`** as the PIE-safe toggle; Escape works in `-game` and packaged runs (Android uses Back).

**Exec commands** (first encounter). `UFUNCTION(Exec)` on a PlayerController is callable from the
console (`` ` `` / `~` on desktop). It's the debug path that still works when the UI is broken, and

controller gets the full set: `EOSLogin`, `EOSLogout`, `EOSHost`, `EOSJoin`, `EOSLeave`, `EOSStatus`,

**Why not PIE, and what replaces it.** One process = one EOS login, so multi-client PIE **cannot**
host-and-join over EOS. The replacement is two **separate processes**, each auto-logging-in as a
different Epic account via the **EOS Dev Auth Tool**: you sign two accounts into the tool once,

`-AUTH_TYPE=developer -AUTH_LOGIN=localhost:8081 -AUTH_PASSWORD=CredN`. The subsystem's `Login()`
already branches on `-AUTH_TYPE` and calls `AutoLogin()`
([EOSSessionSubsystem.cpp:52-58](../UnifiedGameProject/Plugins/OnlineCore/Source/OnlineCore/Private/EOSSessionSubsystem.cpp#L52-L58))
— nothing to write. **The tool is not on this machine and does not ship with the engine** (checked

`SDK\Tools\EOS_DevAuthTool-win32-x64-*.zip`. Start that download before you read further.
