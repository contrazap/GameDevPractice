# P0.2 ‚Äî EOS Plugin Port (guided-build)

Concepts live in the **[P0.2 primer](P0.2-primer.md)** ‚Äî plugin anatomy, the two-level dependency
model, subsystem auto-registration, the what-moves/what-stays table, config layering, the EOS flow
recap, and the DevAuthTool testing model. This plan doesn't repeat them. Source material lives
**in this repo** at [reference/eos/](../reference/eos/) ‚Äî the archival snapshot of the EOS test
project: guide chapters, the verified-facts reference, the secrets pattern, and `src-snapshot/`
holding the exact files this plan ports (`EOSSessionSubsystem.h/.cpp`, 616 lines incl. the voice
workaround, PlayerController, Build.cs, config). The original `MultiplayerTestEOS` project is
provenance only ‚Äî nothing below requires it to exist.

## Goal & design rationale
Port the proven EOS stack (Epic login, host/find/join/leave lobby sessions, RTC voice with
mute/unmute + receive-bug workaround) out of the test project's game module into
`Plugins/OnlineCore/` in `UnifiedGameProject` ‚Äî headless, game-agnostic, small public API ‚Äî then
re-verify Windows‚ÜîAndroid session + voice. This plugin ships in every production game (P0.6 adds
cloud saves to it), so the separation discipline *is* the feature. See primer for full rationale.

## Decisions locked for this plan (approved 2026-07-11)
- **Plugin name `OnlineCore`**, single Runtime module `OnlineCore`, API macro `ONLINECORE_API`
  (name verified free against installed 5.8 engine plugins/modules).
- **API shape:** map URLs as function parameters; bucket ID + defaults via a `UDeveloperSettings`
  class (`UOnlineCoreSettings`); state-change delegates only if time allows (cut line 1).
- **Portal:** the dedicated **UnifiedGameProject** product (same Epic org as GTFOClone ‚Äî existing
  member invites apply). `DefaultArtifactName=UnifiedGameProject`, bucket `UnifiedGameProject_v1`.
  The IDs/ClientSecret are deliberately **not** embedded in this committed plan ‚Äî they live in the
  Epic Dev Portal and your gitignored platform inis only (see step 6).
- **Android re-verify happens in P0.2** (phase entry mandates it; the toolchain on this machine
  already packaged the test project). Full Android pipeline work (device profiles, scalability,
  perf baseline) stays P0.5.

## Scope
**In:** portal pre-flight verification of the new UnifiedGameProject product; the plugin
(descriptor, Build.cs, subsystem port, API cleanup incl. mic-permission move);
project EOS config + secrets templates; minimal game-side session UI (exec commands + C++ widget
base + `WBP_SessionMenu` in UMG ‚Äî login/host/join/leave, mute, resume, quit) calling only the
public API; DevAuthTool two-instance Windows verify; packaged
Win‚ÜîAndroid session + voice re-verify; plugin README (drop-in contract).
**Out:** friends/invites, session browser UI (P1.11); cloud saves (P0.6); any gameplay or the real
character (P0.4); dedicated servers (P5.4); Android device profiles/perf (P0.5).
**Cut lines (drop in order if 3 sessions bust):** (1) state-change delegates ‚Äî poll like the test
did; (2) mute/resume/quit menu buttons ‚Äî keep login/host/join/leave + exec-command mute; (3) if
Android packaging fights back
> ¬Ω session, gate P0.2 on packaged **Win‚ÜîWin** and run the Android re-verify as the opening act of
P0.5 ‚Äî a documented deviation from the phase entry, not silent slippage.

## Multiplayer design (mandatory)
- **Authority model:** listen server ‚Äî the host owns the session lifecycle (create/destroy); joiners
  are clients traveling to the host's resolved EOS P2P address. No dedicated server this phase.
- **What replicates:** nothing new ‚Äî this is the *sessions* layer. No custom replicated properties
  or RPCs are added; connected pawns (engine `DefaultPawn` until P0.4) replicate via engine defaults
  and serve as the visual connectivity proof.
- **Client prediction:** n/a (no gameplay code).
- **Smoke test:** the standard 2-client PIE smoke **cannot** exercise EOS (one process = one login);
  the documented replacement is the two-standalone-instance DevAuthTool run in step 9. What it must
  show: both instances logged in as different accounts, join completes, `EOSStatus` reports
  NetMode Client/ListenServer respectively, pawn movement visible cross-instance, leave returns each
  side to a standalone map without crashes/ensures.

## Mobile & performance (mandatory)
- **Budget rows:** none apply ‚Äî no 3D content is authored (tri/texture/material/light budgets
  untouched). The UMG menu is draw-call-trivial but doubles as the Android session UI ‚Äî buttons
  must be finger-sized.
- **Expected costs:** EOS SDK ships in-engine for arm64 (`libEOSSDK.so`); RTC voice CPU at 2‚Äì4
  players is negligible. Development APK will be large (~1 GB class) ‚Äî irrelevant until P0.5 sizes
  packaging properly.
- **Risks & fallbacks:** Android login needs the browser round-trip (intent filter `eos.<clientid>`
  auto-injected at package time ‚Äî wrong/missing Artifacts line breaks it, check the manifest);
  mic permission must be granted or voice silently sends nothing (plugin now requests it ‚Äî verify
  the dialog appears on first run); both test accounts must remain accepted org members (brand
  review gate ‚Äî the new product shares GTFOClone's org, so existing invites apply). Fallback per cut line 3: gate on Win‚ÜîWin, open P0.5 with the Android run.
