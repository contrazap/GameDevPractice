## Why & what

Port the proven EOS stack (Epic login, host/find/join/leave lobby sessions, RTC voice with
mute/unmute + receive-bug workaround) out of the test project's game module into
`Plugins/OnlineCore/` in `UnifiedGameProject` ‚Äî headless, game-agnostic, small public API ‚Äî then
re-verify Windows‚ÜîAndroid session + voice.

Why it matters beyond practice:
- **Every production game ships this plugin.** Coop + episodic DLC + cross-game progression means
  sessions/login/voice are shared substrate ‚Äî build once, drop into each title. P0.6 adds EOS cloud
  saves to the *same* plugin, so it's scoped account-level, not session-only (hence `OnlineCore`,
  not `SessionCore`).
- **The test project proved the tech but welded it in** ‚Äî hardcoded travel map, hardcoded bucket,
  UI concerns nearby. Porting it properly is the code-organization rep this feature exists for.
- **Second EOS exposure = retention.** Re-walking login ‚Üí lobby ‚Üí P2P travel ‚Üí voice while
  restructuring it is deliberate spaced repetition.

## Mental model

The five ideas that organize everything here:

1. **A plugin is a folder with a passport.** `Plugins/OnlineCore/` + `OnlineCore.uplugin` (JSON
   descriptor). The engine *discovers* it ‚Äî there is no IDE "add project" step; VS solutions are
   generated output. Anatomy in Session 1 Teach.
2. **Dependencies are declared twice, at two levels.** `.uplugin ‚Üí Plugins[]` = plugin graph
   ("when I'm enabled, enable these"); `Build.cs ‚Üí Public/PrivateDependencyModuleNames` = linker
   graph ("my C++ links these"). Miss the first ‚Üí module may not be loaded; miss the second ‚Üí
   linker errors. Public only if the dependency's types appear in your public headers.
3. **Subsystems self-register.** `UEOSSessionSubsystem : UGameInstanceSubsystem` is auto-discovered
   in any loaded module when the GameInstance initializes ‚Äî zero glue code, which is what makes
   "drop the folder into any project" literally true. The game reaches it via
   `GetGameInstance()->GetSubsystem<UEOSSessionSubsystem>()`.
4. **The port is a separation exercise.** Sessions/voice/mic-permission ‚Üí plugin; maps, bucket,
   UI ‚Üí game (parameters + `UOnlineCoreSettings`). The what-moves table (Session 1 Teach) is the
   core [You] thinking.
5. **EOS flow, second exposure:** `Login()` ‚Üí Epic account portal / `persistentauth` cache /
   `-AUTH_TYPE=developer` + DevAuthTool ‚Üí `HostSession()` creates an EOS **lobby**
   (`bUseLobbiesIfAvailable`, bucket attribute, RTC voice room auto-attached) +
   `ServerTravel(map?listen)` ‚Üí clients `FindSessions` (`SEARCH_LOBBIES` + same bucket) ‚Üí
   `JoinSession` ‚Üí `ClientTravel` to the resolved `EOS:<PUID>:GameNetDriver:‚Ä¶` P2P address. Voice
   joins automatically; the workaround re-applies receive options (`SetPlayerVolume(name, 1.0)`)
   after join because 5.8's `EOSVoiceChat` leaves late joiners unsubscribed. Full detail:
   [EOS_Setup_Guide](../reference/eos/EOS_Setup_Guide/) ch. 03/09/10 + the verified-facts table in
   [EOS_MULTIPLAYER_SETUP.md](../reference/eos/EOS_MULTIPLAYER_SETUP.md) ¬ß1.


## Session 1 ‚Äî plugin skeleton & the port

### Teach

**Plugin anatomy.** A plugin = folder under `UnifiedGameProject/Plugins/` + `.uplugin` descriptor +
its own `Source/`, optionally `Content/`, `Config/`, `Resources/`. Target layout:

```
Plugins/OnlineCore/
‚îú‚îÄ OnlineCore.uplugin                descriptor: modules + plugin deps + platforms
‚îú‚îÄ README.md                         drop-in guide (Session 3 ‚Äî the reusability contract)
‚îî‚îÄ Source/OnlineCore/                ‚Üê folder name must equal the module name
   ‚îú‚îÄ OnlineCore.Build.cs            module deps (link level)
   ‚îú‚îÄ Public/                        the API the game may include ‚Äî keep SMALL
   ‚îî‚îÄ Private/                       everything else (module impl, workaround guts)
```

Descriptor fields, verified against the engine's own
`C:\Program Files\Epic Games\UE_5.8\Engine\Plugins\Online\OnlineSubsystemEOS\OnlineSubsystemEOS.uplugin`:

| Field | What it does |
|---|---|
| `FileVersion` | **Mandatory, value 3** ‚Äî descriptor parsing hard-fails without it (`PluginDescriptor.cpp` reads it first and rejects the file otherwise) |
| `Modules[]` | The plugin's code modules: `Name`, `Type` (`Runtime` for us), `LoadingPhase` (`Default` ‚Äî subsystems register at module load), `PlatformAllowList` (`["Win64","Android"]` ‚Äî other platforms skip compiling it) |
| `Plugins[]` | Plugin-to-plugin dependencies: `OnlineSubsystem`, `OnlineSubsystemUtils`, `OnlineSubsystemEOS`, `VoiceChat` |
| `EnabledByDefault`, `Category`, `Description` | Metadata; keep `EnabledByDefault: false` ‚Äî enable per-project in the `.uproject` (the reusable-plugin contract) |

**Names must agree.** Module `"Name"` in the `.uplugin` = `Source/<Name>/` folder =
`<Name>.Build.cs` filename = `class <Name> : ModuleRules` inside it = second argument of
`IMPLEMENT_MODULE`. Any mismatch ‚Üí module not found, or a name-mismatch assert at startup.

**How creation actually works (no wizard, no VS "Add Project").** Files on disk, then: close the
editor ‚Üí create/edit files ‚Üí right-click `UnifiedGameProject.uproject` ‚Üí *Generate Visual Studio
project files* ‚Üí build Development Editor | Win64 ‚Üí open editor. The wizard (Edit ‚Üí Plugins ‚Üí +
Add) would scaffold all of this ‚Äî deliberately unused; the anatomy is the learning. Build-order
gotchas: an **empty/invalid Build.cs breaks project-file generation** (don't regenerate before
step 2 is written), and UBT only compiles **enabled** plugins ‚Äî with `EnabledByDefault: false`
nothing builds until the `.uproject` lists it, so that one line gets pulled forward into step 3.

**The module class ‚Äî what `IMPLEMENT_MODULE` is.** Each module is a DLL in editor builds and
statically linked in packaged builds. `FModuleManager` needs a factory it can invoke by name;
`IMPLEMENT_MODULE(FOnlineCoreModule, OnlineCore)` emits the correct registration for **both** link
modes and static-asserts the name matches the module. Exactly one per module, in a `.cpp`. Our
subsystem self-registers (Mental model #3), so empty lifecycle bodies are *correct*, not lazy ‚Äî
the class exists as the place P0.6 will later hang startup work. Complete skeleton:

```cpp
// Public/OnlineCore.h
#pragma once
#include "Modules/ModuleManager.h"

class FOnlineCoreModule : public IModuleInterface
{
public:
    virtual void StartupModule() override;
    virtual void ShutdownModule() override;
};

// Private/OnlineCore.cpp
#include "OnlineCore.h"

void FOnlineCoreModule::StartupModule() {}
void FOnlineCoreModule::ShutdownModule() {}

IMPLEMENT_MODULE(FOnlineCoreModule, OnlineCore)
```

No `ONLINECORE_API` on the module class ‚Äî nothing outside the module constructs it; the macro
starts mattering on the subsystem (step 3). Engine examples: named-class shape ‚Üí
`C:\Program Files\Epic Games\UE_5.8\Engine\Plugins\Online\EOSShared\Source\EOSShared\Private\EOSSharedModule.h`;
3-line stock-impl alternative ‚Üí
`C:\Program Files\Epic Games\UE_5.8\Engine\Plugins\Online\OnlineSubsystemUtils\Source\OnlineBlueprintSupport\Private\OnlineBlueprintSupportModule.cpp`.

**From `#include` to module name ‚Äî how a Build.cs is actually derived (step 2's real skill).**
An include list alone won't tell you the modules; you must resolve each header to its **owner**:

1. Collect includes from **both** the `.h` and the `.cpp` ‚Äî they land in different lists.
2. Locate each header on disk: in VS, cursor on the `#include` ‚Üí **F12** (or right-click ‚Üí *Open
   Document*); the opened tab's path is the answer.
3. Read the module off the path ‚Äî Unreal's folder layout *is* the module registry:

| Path shape | Module |
|---|---|
| `Engine/Source/Runtime/<M>/Public\|Classes/‚Ä¶` | `<M>` |
| `Engine/Plugins/‚Ä¶/<Plugin>/Source/<M>/Public/‚Ä¶` | `<M>` |
| `Engine/Plugins/‚Ä¶/<Plugin>/Source/Public/‚Ä¶` | the plugin's single module |

**Public vs Private.** Included by *our* public header ‚Üí it's part of our API, downstream code must
resolve those types ‚Üí `PublicDependencyModuleNames`. Included only in `.cpp`/private code ‚Üí
`PrivateDependencyModuleNames`. Over-declaring Public isn't a build break; it leaks our
dependencies' include paths onto the game module.

**Two deps no `#include` reveals:** `CoreUObject` (nothing names it ‚Äî `UCLASS`/`GENERATED_BODY`/
`.generated.h` *are* its machinery; reflection markup ‚áí CoreUObject) and `AndroidPermission` (not in
this file at all ‚Äî it arrives with the mic-permission code moving in from the test's
PlayerController; audit **all code that will live in the module**, not just the file you're reading).

Resolved for this port (`OnlineSessionNames.h` is the one that explains `OnlineBase`;
`IOnlineSubsystemEOS.h` is confirmed a **Public** header of `OnlineSubsystemEOS`, so the voice

**What moves, what stays (the core [You] thinking for steps 3‚Äì4):**

| Concern | Where | Why |
|---|---|---|
| Login/logout, host/find/join/leave, state queries | **Plugin** (public API) | The reusable substrate |
| Voice mute/unmute + RTC receive-bug workaround | **Plugin** | EOS-layer, not gameplay |

**Config as API ‚Äî `UDeveloperSettings` (step 4).** The bucket ID and return map are *the game's*
choices, so the plugin must not hardcode them ‚Äî but it can't take them as function arguments either
(nothing would pass them). `UDeveloperSettings` is the engine's answer: a UObject whose properties
appear automatically in **Project Settings** and persist to an ini. Like subsystems, it
self-registers ‚Äî declare the class and it shows up, no glue. Skeleton:

```cpp
// Public/OnlineCoreSettings.h
#pragma once
#include "Engine/DeveloperSettings.h"
#include "OnlineCoreSettings.generated.h"

UCLASS(config = Game, defaultconfig, meta = (DisplayName = "Online Core"))
class ONLINECORE_API UOnlineCoreSettings : public UDeveloperSettings
{
	GENERATED_BODY()
public:
	/** Partitions EOS lobbies ‚Äî only sessions sharing this bucket are found. */
	UPROPERTY(EditAnywhere, config, Category = "Session")
	FString BucketId = TEXT("UnifiedGameProject_v1");

	UPROPERTY(EditAnywhere, config, Category = "Session")
	FString DefaultReturnMap = TEXT("/Game/Maps/L_Sandbox");
};
```

Read it anywhere with `GetDefault<UOnlineCoreSettings>()->BucketId`. Four traps:
- **`config` appears twice** ‚Äî on the `UCLASS` (`config = Game`) *and* on every `UPROPERTY` that
  must persist. Omit it on a property and the value silently resets each run.
- `defaultconfig` writes to `Config/DefaultGame.ini`, section `[/Script/OnlineCore.OnlineCoreSettings]`
  ‚Äî the pattern is `/Script/<ModuleName>.<ClassName without the U>`.
- **`Build.cs` must gain `DeveloperSettings`** (public ‚Äî the base class is in *our* public header).
  Nothing else pulls it in; without it you get an unresolved-include error.
- Engine example:
  `C:\Program Files\Epic Games\UE_5.8\Engine\Source\Runtime\Engine\Classes\Engine\AssetManagerSettings.h:66`.

**The mic-permission move (step 4).** Source to lift, with its guard:
[MultiplayerCoursePlayerController.cpp:28-30, 58-64](../reference/eos/src-snapshot/MultiplayerCoursePlayerController.cpp) ‚Äî

```cpp
#if PLATFORM_ANDROID
#include "AndroidPermissionFunctionLibrary.h"    // top of file, inside the guard
#endif
...
#if PLATFORM_ANDROID
	if (!UAndroidPermissionFunctionLibrary::CheckPermission(TEXT("android.permission.RECORD_AUDIO")))
	{
		UAndroidPermissionFunctionLibrary::AcquirePermissions({ TEXT("android.permission.RECORD_AUDIO") });
	}
#endif
```

That `#if PLATFORM_ANDROID` is exactly why `AndroidPermission` is a **private, Android-only** dep in
the Build.cs ‚Äî on Win64 the code doesn't exist, so the module isn't linked. It belongs in the
plugin's voice-setup path (it exists *because of* EOS voice), not in a PlayerController's `BeginPlay`.

**Config & secrets (steps 5‚Äì6, the P0.1 pattern activates).** The plugin carries *code*; EOS still
needs per-project engine config that cannot live inside it: `DefaultPlatformService=EOS`, the
`NetDriverEOS` definitions, `[/Script/OnlineSubsystemEOS.EOSSettings]`. Split: **committed**
`Config/DefaultEngine.ini` (all non-secret blocks) vs **gitignored**
`Config/Windows|Android/<Platform>Engine.ini` (the `+Artifacts=(ClientId=‚Ä¶,ClientSecret=‚Ä¶)` line;
committed `*.template` copies document the shape). Platform inis layer over Default and **do** cook
into packaged builds ‚Äî that's why the pattern works. One field the portal does **not** issue:
`ClientEncryptionKey` ‚Äî self-generated 64 hex chars, used to client-side-encrypt P0.6's cloud
saves. Nothing in P0.2 uses it, but it must stay **stable for life** (a changed key makes uploaded
saves undecryptable; everything else keeps working). Generate once, store in the password manager.

## Multiplayer design (mandatory)
- **Authority model:** listen server ‚Äî the host owns the session lifecycle (create/destroy); joiners
  are clients traveling to the host's resolved EOS P2P address. No dedicated server this phase.
- **What replicates:** nothing new ‚Äî this is the *sessions* layer. No custom replicated properties
  or RPCs; connected pawns (engine `DefaultPawn` until P0.4) replicate via engine defaults and are
  the visual connectivity proof.
- **Client prediction:** n/a (no gameplay code).
- **Smoke test:** the standard 2-client PIE smoke **cannot** exercise EOS (one process = one
  login); the documented replacement is Session 2's two-standalone-instance DevAuthTool run. It must
  show: both instances logged in as different accounts, join completes, `EOSStatus` reports
  NetMode Client/ListenServer respectively, pawn movement visible cross-instance, leave returns
  each side to a standalone map without crashes/ensures.

  doubles as the Android session UI ‚Äî buttons must be finger-sized.
